HRS SECURE ARCHITECTURE

Архитектура защищённого стенда

Логическая схема HRS: внешний сегмент, NGFW на OPNsense, TS_DMZ через Tailscale, WEB+DB сервер и локальная PostgreSQL. Tailscale используется как защищённый транспорт, а контроль доступа и сегментация остаются на стороне NGFW. Статус NGFW на этой странице проверяется live-check запросом до адреса шлюза NGFW.

Логические зоны

Разделение инфраструктуры на зоны безопасности.

WAN / External

Внешняя сеть и пользовательские подключения. Прямой входящий доступ ограничивается правилами NGFW. Из-за CG-NAT транспорт реализован через Tailscale.

OPNsense NGFW

Центральная точка фильтрации. OPNsense отвечает за маршрутизацию, правила межсетевого экрана, разделение зон и контроль доступа.

TS_DMZ

DMZ-сегмент поверх Tailscale. WEB-сервер доступен через защищённый туннель, но политики доступа задаются на NGFW.

HRS-WEB

Сервер WEB+DB. На нём работают nginx, Flask/Gunicorn, PostgreSQL и механизмы защиты SSH через Fail2Ban.

PostgreSQL

База данных приложения. PostgreSQL привязан к 127.0.0.1:5432, поэтому не доступен напрямую из WAN, LAN или TS_DMZ.

Fail2Ban-ready

Планируемый механизм защиты SSH от brute-force атак. Срабатывания могут быть отражены в HRS SOC как инциденты безопасности.

Поток данных приложения

Как пользовательский запрос доходит до базы данных.

1. Пользователь открывает WEB-интерфейс
Запрос приходит на nginx по HTTP.
2. nginx передаёт запрос во Flask/Gunicorn
Reverse proxy отправляет запрос на 127.0.0.1:8000.
3. Flask выполняет бизнес-логику
Приложение обрабатывает создание, изменение, удаление и поиск инцидентов.
4. PostgreSQL хранит данные
SELECT / INSERT / UPDATE / DELETE выполняются через локальное подключение 127.0.0.1:5432.
5. Ответ возвращается пользователю
Данные из БД отображаются в интерфейсе HRS SOC Command Center.
